# 1. イントロ — パーサが「正しい」とは何か

[English](../en/01-intro.html) | **日本語** ｜ [目次](../index.html) ｜ [2章 →](02-lean-primer.html)

## 1.1 テストが通ることと、正しいことの間

まず 1 つ、実話から始めます。

このプロジェクトには、パーサ・型検査器・インタプリタを 2 つの実装（Lean と、
そこから生成した Scala）で走らせて突き合わせる差分テストが 60 ケースあります。
全部グリーンでした。ところが開発の最終盤、「印字したプログラムをパースし直すと
元に戻る」という定理を証明しようとしたら、**証明がどうしても通らないケース**が
1 つ見つかりました。調べると、それは証明の技術的な困難ではなく、
**文法そのものに実在する境界条件**でした。関数定義の本体が裸の変数で終わり、
直後の式が `(` で始まると、PEG の優先選択が関数の境界を越えて両者を
「関数呼び出し」として読んでしまう——60 ケースのテストが一度も踏まなかった形です
（詳細は [4 章](04-shallot.html)）。

テストは「試した入力については正しい」ことしか言えません。では
「**すべての**入力について正しい」と言いたければ、どうすればいいのか。
それが**機械検証された証明**であり、このプロジェクトが PEG パーサに対して
実際にやってみたことです。

## 1.2 パーサの「正しさ」を分解する

「このパーサは正しい」を、実装者の日常語で分解してみましょう。

- **健全性（soundness）** — パーサが返した結果（成功も失敗も）は、文法の
  仕様が定める通りか。「受理したものは本当に文法が認めるものか」
- **決定性（determinism）** — 同じ入力に対して、仕様上の結果はただ一つか。
  構文木も含めて一意か（曖昧性の不在）
- **完全性（completeness）** — 仕様が結果を定める入力を、実装は必ず
  処理しきるか。「仕様は認めるのに実装が返さない」が起きないか
- **roundtrip** — AST を印字してパースし直したら、元の AST に戻るか

このプロジェクトでは、この 4 つすべてが Lean 4 の定理として証明されています。
しかも証明の対象は Ford (POPL 2004) の PEG 形式意味論——この文書の読者なら
一度は読んだ、あの推論規則たちです。

## 1.3 プロジェクトの全体像

登場人物は 2 つです。

**Shallot** は int と bool を持つ小さな第一階関数型言語で、その処理系
一式——PEG パーサ、型検査器、インタプリタ、定数畳み込み最適化器、
スタック VM コンパイラ——が Lean 4 で書かれ、それぞれの正しさが証明されて
います。証明の穴（`sorry`）はゼロ、全定理の公理依存はビルド時に機械検査
されます。

**Lens** は Lean 4 のコードを Scala 3 に変換する自作の抽出器です。証明した
処理系を JVM で動くコードとして持ち出します。`shallot-cli` という CLI から
実際に `.shl` ファイルを実行できます——そのパースは形式検証済みの PEG
インタプリタ、型検査は健全かつ完全と証明された検査器です。

全体の流れはこうなっています：

```
lean/Shallot（仕様＋実装＋証明）
   │  lake build = 全証明の検査＋公理監査
   │
   ├─ lake exe extract（Lens 抽出器）──→ scala/generated（約1,300行の Scala 3）
   │                                        │
   │                                        └─→ shallot-cli（run / eval / dump）
   │
   └─ shallot-runner ──┐
                       ├──→ 差分ハーネス（60ケース、Lean 実行 ≡ Scala 実行）
   shallot-cli dump ───┘
```

そして本プロジェクトのいちばん美しい構図はここです：**Shallot のパーサは、
検証済みの汎用 PEG インタプリタに文法データを渡しているだけ**なのです
（`lean/Shallot/Syntax/TreeToAst.lean`）：

```lean
/-- The Shallot parser: the VERIFIED generic PEG interpreter applied to
`shallotGrammar`, then tree extraction. -/
def parseShallot (fuel : Nat) (s : String) : Except ParseErr Program :=
  match pegRun shallotGrammar fuel (.nt NT.program) s.toList with
  | none => .error .fuelOut
  | some .fail => .error .syntaxErr
  | some (.ok (.nodeNT _ t) _) => treeToAst t
  | some (.ok _ _) => .error .shape
```

パーサ生成器がコードを吐く代わりに、**定理が文法値に継承される**。
汎用インタプリタ `pegRun` について証明した健全性・完全性・決定性は、
どんな文法値に対しても成り立つので、`shallotGrammar` に対しても自動的に
成り立ちます。「検証済みパーサ」とは、特別に固いコードのことではなく、
この継承の構図のことなのです。

## 1.4 この連載の読み方

- **[2 章](02-lean-primer.html)** で Lean の読み方を最小限インストールします。
  書けるようになる必要はありません。「inductive は BNF、帰納的述語は推論規則」
  という 2 つの読み替えがすべてです
- **[3 章](03-peg-semantics.html)** が中核です。Ford の意味論が Lean でどう
  書かれ、何が証明されたか。PEG を知っていれば全部読めます
- **[4 章](04-shallot.html)** は言語処理系の証明スタック。冒頭の「証明が
  見つけた文法の穴」の顛末もここです
- **[5 章](05-lens.html)** は Lean → Scala 抽出と、「どこまで信じられるか」の
  正直な線引き
- **[6 章](06-reading-guide.html)** はソースコードの歩き方です

ひとつ先回りしておくと、形式検証は「バグゼロの保証書」ではありません。
証明が守る範囲と、信頼に頼る範囲（TCB）の線引きは 5 章で正面から扱います。
その線引きを含めて、「パーサを証明するとはどういう営みか」の実感を
持ち帰ってもらえたら、この連載は成功です。
