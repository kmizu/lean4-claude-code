# 7. 応用編 — 検証済み JSON パーサを作る

[English](../en/07-json.html) | **日本語** ｜ [← 6章](06-reading-guide.html) ｜ [目次](../index.html) ｜ [8章 →](08-macro-peg.html)

6 章の最後に「この土台の上でやれる未踏の題材はたくさん残っている」と書きました。
この章はその最初の回答です：3 章の PEG フレームワークの上に、**実用フォーマットの
検証済みパーサ**——RFC 8259 の JSON——を 1 本立てました。汎用フレームワークへの
投資がどれだけ回収できるかの実地報告でもあります。

## 7.1 文法を書く＝定理を継承する

やることの第一歩は、RFC 8259 の ABNF を `PExp` の値に写すことだけです
（`lean/Json/Grammar.lean`）。RFC は空白の置き場所まで ABNF で規定しているので
（`begin-object = ws %x7B ws` — 空白は構造トークンの内側）、それをそのまま
写します。ファイルのヘッダに ABNF との 1:1 対応表を置きました。

そして 1 章の構図がそのまま効きます：**パーサは検証済みの汎用 `pegRun` に
この文法値を渡すだけ**なので、健全性（T1）・木込みの決定性（T2）・完全性（T3）は
書いた瞬間に JSON パーサの定理です。追加の証明はゼロ行。

## 7.2 AST の設計：構文に忠実であること

面白いのはむしろ AST の設計判断でした（`lean/Json/Ast.lean`）。

**数値は構文のまま保持します。** RFC 8259 は数値の内部表現を意図的に規定して
いません。パース時に浮動小数点数へ潰すのは実装判断の混入なので、符号・整数部・
小数部・指数部を**桁の列のまま**持ちます。おかげで `1E5` と `1e5`、`1e+5` と
`1e5` の区別まで保存され、roundtrip 定理が「正規化を除いて」のような但し書き
なしの**厳密な等式**になります。実は開発中、印字の固定点検査（`#guard`）が
「大文字 `E` が小文字化して返ってくる」ことを検出して、指数部の大文字小文字を
保持するフィールドを AST に足させました——ビルド時テストが設計を修正させた
一幕です。

**文字列はデコード済みのコードポイント列です。** `\uXXXX` エスケープは
サロゲートペアの合成まで含めてパース時に解決します。孤立サロゲートは
**拒否**します（RFC は実装定義としているので、保守的な選択を採り、
ソースコードに設計判断として明記しました）。

## 7.3 実戦テスト：JSONTestSuite

形式検証と経験的テストは対立しません。JSON パーサの適合性テストのデファクトである
nst/JSONTestSuite（318 ファイル）を両側のランナーで走らせます：Lean ネイティブの
検証済みパーサと、Lens で抽出した Scala 版です。結果は：

- **y_（必ず受理すべき）違反：0 件。n_（必ず拒否すべき）違反：0 件。**
- i_（実装定義）は孤立サロゲート拒否の方針どおり 11 受理 / 24 拒否
- **Lean 版と抽出 Scala 版の判定は 318 ファイル全件で同一**
  （不正 UTF-8 バイト列の扱いまで揃えてあります）

RFC の ABNF を素直に写した文法が初回からこの成績だったのは、「仕様を仕様のまま
書ける」ことの威力だと思います。このチェックは `make verify` に常設され、
以後のすべての変更に対して回り続けます。

## 7.4 roundtrip 定理

締めは 4 章と同じ形の定理です（`lean/Json/Roundtrip.lean`、1,367 行）：

```lean
theorem parse_print_json (v : JValue) (hwf : wfValue v = true) :
    ∃ fuel, ∀ fuel', fuel ≤ fuel' → parseJson fuel' (printJson v) = .ok v
```

どんな well-formed な JSON 値も、正準印字して検証済みパーサに読み戻させると
**ちょうど元の値**に戻る。注目してほしいのは仮定 `wfValue` の軽さです。
中身は「数値の桁が RFC の形をしている」だけ——**文字列には仮定が一切
ありません**。文字列はコードポイント列という型そのものが整形性を保証し、
エスケープ処理の 4 分岐（引用符・バックスラッシュ・制御文字・素通し）が
文法の char 規則の解析形と 1:1 に対応することを証明したからです。

Shallot の roundtrip（4 章）では PEG の境界条件が見つかって分離ガードが
必要になりましたが、JSON では**そうした罠が出ませんでした**——区切り記号が
すべて明示的な文法は、こういうところで強いのです。これも証明を書いて初めて
断言できることですが。

## 7.5 回収できた投資

このパーサの新規記述は、文法データ・AST・木の抽出器・printer・整形述語・
字句補題・roundtrip 証明を合わせて約 2,600 行。対して**再利用**したのは、
PEG の意味論と 4 定理、導出構築のパターン、抽出器 Lens 一式、差分ハーネスの
機構、そして証明の設計語彙（燃料規約・正準印字・fail-loud）のすべてです。
2 本目のパーサはこんなに安い——汎用フレームワークを証明付きで 1 度作る
ことの意味は、ここにあります。

JVM から使うなら：

```sh
cd scala
sbt "shallotCli/run json '{\"a\": [1, 2.5e3]}'"   # => ok:{"a":[1,2.5e3]}
```

このパースを実行しているのは、健全性・完全性・決定性・roundtrip が
証明された、Lean から抽出されたコードです。
